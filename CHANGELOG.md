# Changelog

This repo releases via the `release-same-version` label (see `.github/workflows/publish.yml`): every package in the workspace ships the same version bump per release, even packages without code changes. This file exists so consumers can tell a content release from an alignment bump.

Per-package detail lives in the GitHub release tagged `<npm-name>@<version>`.

## Unreleased

### Added — remote-manifest × locally-registered-component pairing

Formalizes the "backend data selects a locally-installed view" join: a remote manifest carries a string **discriminator** id, the component ships as code and registers through the normal module → slot path, and the host pairs the two by id at render time. The helpers are **read-side projections of an already-resolved slot** — pure functions, peers of `mergeRemoteManifests` / `buildSlotsManifest`, not a registration path — so they add no module type and no new ingress, and a Vue `computed` (or a React `useMemo`) re-runs them on reactive change with no library-specific glue.

- **`@modular-frontend/core`** — `resolveComponentRegistry(entries, opts?)` indexes a slot of `ComponentEntry<C, TMeta>` into a `ComponentRegistry` (`get` / `getEntry` / `has` / `ids` / `entries`). Duplicate ids **throw** by default (the same fail-loud stance as duplicate module ids); `onDuplicate: "last-wins"` / `"first-wins"` opt out when a deployment intentionally shadows a first-party id. `pairById(items, registry, idOf)` partitions manifest entries into three explicit buckets — `paired` (id resolved), `missing` (an id was requested but no component is installed — a dangling reference to warn on and fall back for), and `unref` (`idOf` returned `undefined` — no view requested) — so the host handles a dangling reference deliberately instead of a silent miss. The component type `C` is opaque (Vue `Component`, React `ComponentType`, anything), which is what lets the helpers live in the neutral engine and serve every binding. New public types: `ComponentEntry`, `ComponentRegistry`, `OnDuplicateComponentId`.
- **`@modular-frontend/core`** — `componentPairingPlugin({ componentSlot, staticRefs, onDuplicate? })`, a thin resolve-time reference-integrity validator (a peer of `validateNoDuplicateIds`, opt in via `registry.use(...)`). It fails assembly when a **statically-registered** manifest reference names an id with no registered component in the slot, and surfaces duplicate component ids at resolve time too. It sees static slot contributions only — `dynamicSlots` and async remote manifests aren't evaluated at validate time, so `pairById`'s `missing` bucket remains the runtime path for backend-delivered manifests. New public types: `ComponentPairingPluginOptions`, `ComponentRefSpec`.
- **`@modular-vue/core`** — now re-exports the remote-manifest surface (`mergeRemoteManifests`, `RemoteModuleManifest`, `RemoteNavigationItem`, `MergedRemoteManifests`) and the pairing helpers (`resolveComponentRegistry`, `pairById`, `componentPairingPlugin` + their types), so a Vue consumer imports them from the binding it programs against instead of reaching into the neutral engine. (`@modular-react/core` already surfaces the whole engine via `export *`, so it picks these up automatically.) The `@modular-frontend/core` peer range widens to `^0.1.0 || ^0.2.0` so the peer contract admits the engine release that actually ships these symbols.
- **Docs** — `docs/remote-capability-manifests.md` gains a "Pairing wire-safe manifests with code-shipped components" section (the id-namespacing, the join, missing-reference handling), and its two anti-pattern mentions are redrawn to distinguish _conjuring_ a component from a wire string (still forbidden) from _selecting_ among slot-registered, code-shipped components by a data id (supported). Cross-linked from the Nuxt consumer-seam guide.

### Added — journey hosting and URL sync

Closes items 1 and 2 of `docs/consumer-feedback-production-app.md`: the two largest pieces of glue a production consumer wrote _around_ the library (a ~300-line bidirectional URL reconciler, and a host wrapper duplicated per journey) now ship in it. Both land on React and Vue together.

- **`@modular-frontend/journeys-engine`** — `createJourneySync(runtime, instanceId, port, options?)`, the journey ↔ location reconciler. Framework- **and** router-neutral: the location side is a narrow `JourneySyncPort` seam (`read` / `push` / `replace` / optional `go` / `subscribe`) that an app fills in for its router in ~6 lines, and paths are opaque strings the sync only ever compares for equality — so the port owns base paths, segment-vs-search-param, and encoding. The journey advances and the URL follows; the user presses Back or Forward and the journey follows. **Event-sourced, not a diff**: the tempting implementation compares URL to runtime on every render and cannot work, because "the journey advanced" and "the user pressed Back" both present as _the URL naming an earlier step than the runtime_, and telling them apart after the fact needs bookkeeping that races the render loop. Reacting to the runtime's notification and the port's notification as separate signals means the direction is known before anything is decided. Both directions are idempotent, so the sync's own `push` re-enters, resolves to "already there", and stops — no echo-suppression latch for the synchronous case. The one genuinely-async write, the self-rewind `port.go(-n)`, is guarded by a navigation-generation counter: if the runtime advances again before a real router settles the `go`, the late echo is recognised as stale and re-asserts the URL rather than dragging the runtime back over the newer step. Push-vs-replace follows the journey's own depth: advancing pushes (so Back has somewhere to go), an in-journey rewind uses `port.go(-n)` (so the browser's forward stack survives and Forward redoes the step; falls back to `replace` when the port has no `go`), and the first sync replaces (the host's navigation already spent an entry). Also exported: `resolveJourneySyncAction` (the pure decision table — `none` / `rewind` at an index / `forward` by N / `unresolved`, testable with no runtime and no router), `journeyStepPath`, `defaultStepPath`, and `createMemoryJourneySyncPort` (a browser-shaped in-memory port for tests and headless hosts).
- **`@modular-react/journeys`**, **`@modular-vue/journeys`** — `useJourneySync(instanceId, port, options?)`. The React hook / Vue composable lifetime wrapper over `createJourneySync`. Tolerates a `null` id so it can be called before the instance exists, and reads the port and the `stepToPath` / `onUnresolved` / `onBlocked` callbacks through a live ref — an inline port literal will not rebuild the reconciler, which matters because rebuilding re-runs the initial reconcile and that reconcile navigates. The port's `go` is read reactively, since its _presence_ changes the rewind strategy.
- **`@modular-react/journeys`**, **`@modular-vue/journeys`** — `<JourneyHost handle={h} input={i} />` and `useJourneyHost(handle, input, options?)`. Starts the journey on mount, renders its current step, ends **and forgets** the instance on unmount (`<JourneyOutlet>` alone only ends; forgetting is what stops terminal records accumulating in a long-lived shell). The unmount `end` is forced (`{ force: true }`) so a non-terminal `onAbandon` result (`{ next }` / `{ invoke }`) is coerced to an abort rather than leaving an active instance nothing renders — a terminal `onAbandon` choice is still honoured, and because host and outlet both force, `onAbandon` runs once per unmount. Accepts and forwards every outlet prop, and takes a render-prop child / scoped slot receiving `{ instanceId, instance, stepIndex, outlet }` for chrome. The instance is fixed for the host's lifetime — `handle` / `input` are read once at mount, since abandoning a half-finished flow because a prop identity changed is never what the caller meant; remount with a `key` to switch journeys. With a `persistence` adapter, "start" means resume.

  Two behaviors worth knowing. **The journey starts from an effect, not during render**, so `instanceId` is `null` for the first render (`<JourneyHost>` shows `loadingFallback`): `runtime.start()` mutates the runtime and may write to persistence, which a render React discards or replays would duplicate — this is also why the obvious hand-rolled `useState(() => runtime.start(…))` is unsafe under StrictMode: its initializer runs twice and only one result is kept, so without idempotent persistence the discarded `start` leaks an instance (deterministic persistence bounds the damage by converging both calls on the same instance). And **`stepIndex` (`history.length`) ships without a `stepCount`**: a journey's next step is computed by a transition handler from live state, so the total is not knowable from a running instance, and a hand-passed total is exactly the duplicated flow encoding a progress API should be removing. Deriving it from the transition graph is feedback item 4.

  A URL cannot navigate a journey to an arbitrary step, and the API does not pretend otherwise. A step is derived from state and `JourneyStep` carries no identity (`{ moduleId, entry, input }` and nothing more), so the only positions a location can select are ones the journey has already been to (`history` → `rewindTo`) or just rewound from (`future` → `goForward`). Anything else fires `onUnresolved` and navigates nothing — the sync will not drag a user back who clicked a link out of the flow on purpose. A frame the journey refuses to leave (no `allowBack` opt-in, or a child in flight) re-asserts the current step's URL — the standard "block the Back button" shape — and fires `onBlocked`; with `go` it walks the browser forward by the rejected distance so a refused multi-entry jump keeps the intermediate entries, falling back to a (truncating) `push` only when the port has no `go`. A sync tracks the one instance it is given and does not follow child journeys — a child in flight simply reads as a blocked parent frame.

### Added — journey runtime additions (EXP-1848 adoption follow-up)

- **`@modular-react/core`** — `ModuleEntryPoint.buildInput?: (state) => TInput`. When declared on an entry, the journey runtime calls it on every step entry (initial start, forward push, `goBack` pop, resume-into-step) AND on any same-step state change (an `{ invoke }` carrying `state`, or a resume that bumps state without advancing) and uses the result as the step's `input`. Lets a back-navigated form re-render against the journey state accumulated by earlier exits instead of the input frozen at first push. Opt-in — entries without `buildInput` keep the current cache-on-push behaviour. Throws abort the instance with a typed `JourneySystemAbortReason` (`"build-input-threw"`); in `debug` mode a one-time warning fires when the handler's `next.input` would have been overridden by a differing `buildInput` result. Authors annotate the `state` parameter with the hosting journey's `TState` (the module surface stays journey-agnostic).
- **`@modular-react/core`** — `StepSpec`'s `input` field is now **optional** for entries that declare `buildInput`, and stays **required** for entries that don't. Since the runtime re-derives a `buildInput` entry's `input` from journey state and discards whatever a transition stamps, forcing authors to supply a dead value served no purpose. Transition `next` literals, `JourneyDefinition.start`, `defineTransition`, and `selectModule` / `selectModuleOrDefault` all accept the omission. Stamping a real value still type-checks (the runtime discards it and `debug`-warns on drift), so this is a non-breaking widening. `defineEntry` now preserves whether `buildInput` was supplied in its return type (eight overloads — eager/lazy × `mountKinds` × `buildInput`). New public types: `StepInputSlot<TMod, E>` and `EntryAt<TMod, E>`. Note the asymmetry for module authors: _adding_ `buildInput` to a module is a non-breaking widening for downstream journeys, but _removing_ it flips `input` back to required — a deliberate compile error telling journeys that omitted `input` they must supply it again. `JourneyStepFor` (the observation type) is unaffected — the runtime always populates `input`, so it stays required there.
- **`@modular-react/core`** — `JourneyRuntime.goBack(id: InstanceId): void` on the public surface. Equivalent to the `goBack` prop the runtime hands the active step's component, but addressable by instance id so a shell that owns its own back button (browser `popstate`, hardware back, breadcrumb) can wire `popstate → runtime.goBack(id)` directly instead of capturing the active step's callback through a React context. No-op under the same conditions as the closure form (unknown id, terminal / loading, child in flight, no `allowBack`, empty history).
- **`@modular-react/core`** — `JourneyRuntime.canGoBack(id: InstanceId): boolean` and `JourneyRuntime.canGoForward(id: InstanceId): boolean`. Read-only predicates that mirror the guards in `goBack` / `goForward` exactly — same `status` / `activeChildId` / history-or-future-length / `journeyAllowsBack` / `entryAllowBackMode` checks, including the documented module-map fallback (no descriptor + journey opt-in → treat entry as `preserve-state`). Lets a shell-owned Back / Forward toolbar gate enable-state authoritatively from a single instance id instead of duplicating the runtime's transition/entry opt-in logic in shell code. Pure reads — no mutation, no notifies.
- **`@modular-react/core`** — `JourneyRuntime.goForward(id: InstanceId): void` and `JourneyInstance.future: readonly JourneyStep[]`. Inverse of `goBack`: each `goBack` pushes the step it rewinds from (plus the post-transition state and the popped rollback snapshot) onto a per-instance future stack; `goForward` pops it and restores all three so the runtime lands back at the step the user just rewound from. Lets shells wire browser Forward (or a redo button) to journey navigation instead of having the URL drift ahead of the runtime. Does NOT re-run the transition handler (side effects don't double-fire) or `buildInput` (the captured `step.input` was already built against the same state being restored — asymmetric with `goBack`, which re-runs `buildInput` on re-entry). For a `rollback`-mode entry, any edits the user made between the rewind and the redo are discarded. Cleared by any fresh exit-driven transition (matches browser back/forward semantics — `invoke` leaves the stack intact since the parent's step doesn't advance). Mirrored as `ModuleEntryProps.goForward?: () => void` on the per-step props (present only when the future stack is non-empty — most shells will wire Forward at the shell level via the runtime method), and as `simulator.goForward()` / `harness.goForward(id)` for headless testing. Transient: not persisted, reset on hydrate and on `start` recycle. The exposed `instance.future` is bare-step (no internal state / snapshot) so consumers can gate UI on `instance.future.length > 0` without reaching into runtime internals.
- **`@modular-react/journeys`** — `useJourneyInstance(id)`, `useJourneyState<T>(id)`, `useActiveLeafJourneyInstance(rootId)`, and `useActiveLeafJourneyState<T>(rootId)`. React hooks that subscribe to a runtime instance via `useSyncExternalStore`. The `*Instance` variants return the full `JourneyInstance | null`; the state variants are sugar over `instance?.state`. The leaf variants walk `activeChildId` and re-subscribe as the chain grows / shrinks. Prefer the `*Instance` variants when the host needs `step` / `status` / `terminalPayload`, or when the leaf may be any of several journeys (`inst.journeyId` is a natural discriminator).
- **`@modular-react/core`** — new public type `JourneyStepFor<TModules>`. Discriminated union of every concrete `JourneyStep` reachable in a journey's module map; narrowing on `moduleId` + `entry` surfaces the entry's typed `input` without a cast. `JourneyStep` itself becomes `JourneyStep<TInput = unknown>` (backwards-compatible — existing usages default to the wide form). The `simulateJourney` `JourneySimulator<TModules, TState>`'s `step` / `currentStep` / `history` now use the typed union, so tests can assert on per-entry input shapes without `Record<string, unknown>` casts.
- **`@modular-react/journeys`** — `simulateJourney` gains a fourth generic, `TOutput = unknown`. Journeys with a concrete terminal payload type are now assignable to the simulator's parameter without the `as unknown as Parameters<typeof simulateJourney>[0]` cast required by the previous `unknown`-output signature. `SimulateJourneyOptions.modules` lets headless tests bind module descriptors so `buildInput` re-runs at every step entry (without descriptors, the runtime falls back to the cached handler-supplied input — identical to pre-`buildInput` behaviour).

### Added — catalog import resolution

- **`@modular-react/catalog`** — `CatalogConfig.resolve` forwards module-resolution overrides to the harvester's Vite SSR loader. The harvester loads source files with `configFile: false` (so the project's own `vite.config` never leaks into the scan), which previously meant any module importing through a **path alias** (`@app/ui`, `@shared/*`, …) failed to load and silently dropped out of the catalog. `resolve.alias` mirrors those aliases — object form (`{ "@ui": "./packages/ui/src" }`, prefix-matched like Vite; relative replacements anchored to the config dir, absolute paths and bare specifiers passed through) or array form (`[{ find: /^@app\//, replacement }]`). `resolve.dedupe` forwards to Vite's `resolve.dedupe` for singletons (e.g. `react`) pulled in when component files load under SSR. Both are global Vite `resolve` options, so they apply to the SSR load. Also: the harvester now passes `optimizeDeps.entries: []`, stopping Vite's initial dep scan from crawling the project for HTML entry points (the source of harmless-but-noisy "server is being restarted or closed" errors at server close). New public types: `CatalogResolve`, `CatalogAlias`.

### Fixed — catalog dep-scan noise from the config loader

- **`@modular-react/catalog`** — the `optimizeDeps.entries: []` quieting added above only covered the harvester's SSR server; the **config loader** (which `ssrLoadModule`s `catalog.config.ts`) still created a plain Vite server with the optimizer enabled. Loading the config file — which imports `@modular-react/catalog`, a real bare specifier — kicked off the dep scan, and the loader's immediate `server.close()` raced it, so `catalog build` still printed `Failed to scan for dependencies` / `server is being restarted or closed` on every run (most visible under Vite 8.0.12+, where the scan reliably outlives the close). Both loaders now build their server from one shared `createCatalogSsrServer` factory, so the disabled optimizer can't drift back out of either. Output was always correct — this only removes spurious error output. Reproduced and verified end-to-end against a real aliased consumer: zero dep-scan lines on `build`.

### Fixed — journeys: persisted step pointing at a renamed/removed module

- **`@modular-react/journeys`** — a persisted journey whose saved **active step** named a `(moduleId, entry)` that no longer existed (module renamed/removed, or a flow relocated to a different module) used to hydrate that step verbatim — even when the blob's `version` still matched — and `<JourneyOutlet>` rendered its dead-end "no entry on the registered modules" notice with no way forward. The runtime now reconciles the (post-`onHydrate`) blob's active step against the registered modules and treats an unresolvable step like a version mismatch: `runtime.start()` discards the blob via `persistence.remove(key)` and mints a fresh instance under the same key (default — no `onHydrate` required; a one-line warning naming the step fires under `debug`), and `runtime.hydrate()` throws a `JourneyHydrationError` naming the offending step so the persistence-unlinked caller can drop the instance or `start()` fresh. Scoped to the active step (history, which drives `goBack`, is left untouched); a no-op when the runtime has no module map. A migrator (`onHydrate`) runs first, so an app that prefers to preserve in-flight journeys across a rename can rewrite the step rather than restart. Surfaced from a real consumer (lokalise/autopilot#8496), where `legacyIntegrations.projectCreation.sourceSelection` was moved to `project.projectCreation.sourceSelection`.

### Behavior changes

- **`@modular-react/core`** — `buildNavigationManifest` (and therefore `useNavigation`) now breaks ties on `order` by preserving insertion order instead of label string comparison. Items declared first render first when `order` is unset or equal: modules in registration order, items in the order declared in each module's `navigation` array, plugin-contributed items last. Labels are no longer a tiebreaker — the previous fallback sorted by i18n-key name (e.g. `appShell.nav.assets` < `appShell.nav.projects`), which produced surprising orderings unrelated to translated text. Apps that relied (intentionally or not) on alphabetical-by-key fallback should set explicit `order` values to lock in the desired sequence.

### Peer-dep ranges

- **`@modular-react/react`** and **`@modular-react/journeys`** — peer ranges for `@modular-react/core` (and, for journeys, `@modular-react/react`) bumped from `^1.2.0` to `^2.0.0` to match the packages' actual release lines. Drift went unnoticed when `@modular-react/react@2.0.0` and `@modular-react/journeys@1.0.0` shipped: both packages already require a 2.x core at runtime (lazy entry-point support, `EagerModuleEntryPoint | LazyModuleEntryPoint` discriminated union, `resolveEntryComponent` with sync-thenable fast path), but the peer descriptors still pointed at 1.x. Consumers installing `journeys@1.0.0` against `core@^2.0.0` get an `ERESOLVE` error from npm, and `--legacy-peer-deps` workarounds end up dual-installing core (one hoisted 2.x copy plus a nested 1.x copy under journeys), which fragments singletons (`registry`, `defineEntry` identity) at runtime. This is a peer-descriptor correction, not a behavior change.
- **`@tanstack-react-modules/core`**, **`@tanstack-react-modules/runtime`**, **`@tanstack-react-modules/testing`**, **`@react-router-modules/core`**, **`@react-router-modules/runtime`**, **`@react-router-modules/testing`** — same peer-range correction: `@modular-react/core` and `@modular-react/react` bumped from `^1.2.0` to `^2.0.0`; testing packages' peer of `@modular-react/journeys` bumped from `^1.0.0` to `^1.0.1` to match the corrected upstream. Same drift cause and same consumer impact as the previous entry — without this fix, runtime packages nest a 1.x core under a 2.x host, which breaks React Context identity for `useZones` / `useNavigation` / `useRouteData` (the runtime hooks consume contexts created by `@modular-react/react` in the host's `<Manifest.Providers>` tree). Peer-descriptor correction only.
